## D-004: username 변경 불가 (immutable)

**날짜**: 2026-04-20
**버전**: v0.20
**상태**: 적용 중

**결정**: `users.username` (@아이디)은 가입 후 변경 불가.

**대안**:
- 자유롭게 변경 가능 — URL 깨짐, 팔로우 관계 훼손
- 기간 제한 변경 (6개월에 1회 등) — 복잡도 증가
- 어드민 승인 하에 변경 — 운영 부담

**근거**:
- `/creator/@:username` URL에 직접 사용되어 변경 시 외부 공유 링크 깨짐
- 팔로우, 책갈피 등 관계 데이터가 username 기반으로 연결
- 인스타그램, 트위터 등 표준 크리에이터 플랫폼 패턴 따름
- 닉네임은 별도로 자유롭게 변경 가능하므로 유저 표현 자유는 보장

**트레이드오프**:
- 가입 시 고민 없이 정한 아이디를 바꾸고 싶을 때 불만 발생 가능
- → 대신 가입 시 "변경 불가" 안내 문구 명확히 표시

## D-005: 회원가입에 이메일 + @아이디 + 닉네임 3개 요구

**날짜**: 2026-04-20
**버전**: v0.20
**상태**: 적용 중

**결정**: 가입 필수 입력값을 이메일(로그인용, 비공개) + @아이디(공개 식별자) + 닉네임(표시명)으로 분리.

**대안**:
- 이메일 + 닉네임만 (@아이디는 마이페이지에서 설정) — 설정 안 하는 유저 발생
- 닉네임 = @아이디 (통합) — URL에 한글/이모지 사용 불가
- 이메일만 (익명 가입) — 크리에이터 인지도 형성 불가

**근거**:
- 크리에이터 중심 플랫폼으로 포지셔닝하려면 고유 식별자(@아이디) 필수
- 역할 분리: 이메일=로그인, @아이디=공개 식별자, 닉네임=자유로운 표시명
- 인스타그램 표준 패턴 (@username + display name)
- 나중에 설정하게 두면 설정 안 하는 유저 발생 → 가입 시 필수

**트레이드오프**:
- 가입 폼 입력 항목 증가로 이탈률 상승 가능
- → @아이디 필드에 실시간 중복 확인 UI로 이탈 최소화

## D-011: 동적 SQL 쿼리는 db/index.js 헬퍼 함수로 분리

**날짜**: 2026-04-23
**버전**: v0.23
**상태**: 적용 중

**결정**: strftime format·테이블명·컬럼명처럼 SQL 파라미터(`?`)로 처리할 수 없는 동적 값이 필요한 쿼리는, `db.prepare()`를 `db/index.js` 내부의 헬퍼 함수로 분리하고 라우터에서는 그 함수만 호출한다.

**대안**:
1. 라우터에서 `db.prepare()` 직접 호출 허용 (기존 admin.js 방식) — 컨벤션 위반
2. 가능한 조건 조합을 모두 별도 prepared statement로 등록 — 조합 수가 많을 때 비현실적
3. ORM 도입 — 프로토타입 규모에서 오버엔지니어링

**근거**:
- SQLite의 `strftime` 첫 번째 인자(format string)와 테이블명·컬럼명은 SQL 파라미터화 불가
- 라우터에서 `db.prepare()` 호출을 허용하면 컨벤션 경계가 무너짐
- `db/index.js`에 격리하면 SQL이 한 곳에 모이고 라우터는 함수 호출만 담당

**트레이드오프**:
- prepared statement 캐싱 불가 (호출마다 새로 compile) → 어드민 전용 저빈도 쿼리라 성능 영향 미미
- 헬퍼 함수 인자에 테이블명·컬럼명을 그대로 받으므로 SQL injection 주의 필요 → 라우터에서 whitelist 검증 후 전달

## D-012: 보안 프레임워크 일괄 적용 (helmet + rate-limit)

**날짜**: 2026-04-23
**버전**: v0.23
**상태**: 적용 중

**결정**: helmet(보안 헤더), express-rate-limit(속도 제한), cookie secure/httpOnly 플래그를 일괄 적용. csurf는 제외하고 SameSite=lax로 CSRF 방어.

**대안**:
1. csurf 적용 — 2023년 deprecated, 유지보수 중단. SameSite=lax가 동일 기능 커버하므로 제외
2. rate-limit 인메모리 대신 Redis — 프로토타입 규모에서 오버엔지니어링, 단일 서버 배포라 불필요
3. check-username에 authLimiter(10회) 동일 적용 — 실시간 타이핑 UX 파괴, 별도 30회 limiter 분리

**근거**:
- 포트폴리오 제출 전 기본 보안 체계 확립
- helmet은 설정 한 줄로 10+ 헤더를 일괄 적용, 비용 대비 효과 최대
- SameSite=lax는 CSRF의 가장 큰 공격 벡터(cross-origin form POST)를 차단

**트레이드오프**:
- CSP `unsafe-inline` 허용 — 현재 인라인 스크립트/스타일 다수 사용, nonce 방식으로 전환 시 리팩터링 필요
- rate-limit 스토어가 인메모리 — 서버 재시작 시 초기화, 분산 환경 미지원

**대안**:
- React + Vite 즉시 마이그레이션
- Next.js App Router로 전면 재작성
- 점진적 마이그레이션 (화면 단위 React 컴포넌트 교체)

**근거**:
- 포트폴리오 프로토타입 단계에서 마이그레이션 실익 낮음
- 예상 소요: 2–3주 (현실적으론 1달 이상 — 전형적인 과소 추정 패턴)
- 점진적 마이그레이션보다 실사용자 발생 시 Next.js로 처음부터 새로 짜는 게 기술 부채 적음

**트레이드오프**:
- Vanilla JS 유지 → 팀 합류 시 온보딩 비용 증가 가능
- React 즉시 전환 → 현재 작동하는 기능들에 버그 유입 리스크

**마이그레이션 시 핵심 위험 포인트**:
1. **전역 상태 의존성** — `_currentUser`, `currentCharacter` 등 전역 변수가 암묵적으로 공유됨. React Context/Zustand로 전환 시 참조 관계 전수 파악 필요
2. **`innerHTML` 템플릿 리터럴 규모** — app.js 전체에서 HTML을 문자열로 조립. JSX 변환 중 `escapeHtml`, 조건 분기, 이벤트 핸들러 혼재로 버그 유입 가능성 높음
3. **채팅 스트리밍** — `fetch` + `ReadableStream` 직접 파싱. React 상태 업데이트 사이클과 충돌 가능 (`useEffect` 클린업, 언마운트 중 setState 에러)

**마이그레이션 전 준비 작업 (지금부터 가능)**:
1. 전역 상태를 단일 `state` 객체로 통합 → Context 전환 단순화
2. `onclick` 인라인 핸들러 → 함수 분리 (P2 접근성 작업과 병행 가능)
3. `app.js`(4000줄)를 화면 단위 파일로 분리 → React 컴포넌트 파일 구조와 바로 매핑됨

**향후 전환 시 권장 스택**:
- Next.js App Router (점진적 마이그레이션 대신 전면 재작성)
- 상태 관리: Zustand (규모 대비 가벼움)
- 스타일: 기존 CSS 토큰 유지 또는 Tailwind
